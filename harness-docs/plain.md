# Harness 框架：大白话完整版

> 本文目的：让你**完全不依赖任何技术术语**，也能彻底搞懂这套框架。  
> 读完之后，回头看 [design.md](./design.md) 和 [suggestion.md](./suggestion.md) 里的工程化措辞会变得很容易。

---

## 一、它是什么？一句话讲完

想象你新招了一个**远程实习生**。

他很聪明，懂很多——但你不在他旁边，你不知道他啥时候会犯傻：

- 可能把"测试库"和"生产库"搞混
- 可能给客户承诺一个我们做不到的事
- 可能花了 8 小时去搞一个 10 分钟的需求，烧了一堆钱
- 可能默默改了一个文件，没人知道

你要让他**能干活、又不出大事**，会做 **5 件事**。

**这 5 件事就是 harness engineering。** 名字听起来吓人，本质就是这件事。

> "harness" 这个词来自**马具**——给马套上缰绳和挽具，让它有力但不脱缰。  
> AI 也是一样：模型本身像马，再厉害也需要"挽具"才能拉车干活。

---

## 二、那 5 件事是什么？

### 第 1 件：给他一份**工作手册**

> "我们公司是这么做事的。"

里面写：

- 我们的代码是怎么组织的
- 客服该用什么口气
- 哪些话不能说
- 出问题找谁

放在仓库里一个固定位置，实习生（AI）干活前先读一遍。

> 技术文档里叫这个 **Guide / AGENTS.md / 系统提示词**。

---

### 第 2 件：给他**能用的工具，但不是全给**

实习生需要工具：查订单、发邮件、改代码、跑测试……

**但不是所有工具都给：**

- 查询给，**改数据不给**（除非他证明自己靠谱）
- 测试库给，**生产库不给**（除非有人审）
- 常规命令给，**`rm -rf` 这种不给**

而且**每个工具都要有"使用说明"**——AI 不是随便点的，调用前要校验参数对不对。

> 技术文档里叫 **Tool 层 + Safety 层**。

---

### 第 3 件：他每交一次活，**先让"小机器人"检一遍**

实习生交了作业，**别直接用**。先派几个小机器人检：

- "格式对不对？" — 比如代码 lint
- "测试过没？" — 比如跑单测
- "有没有违禁词？" — 比如客服场景
- "退款金额超没超？" — 比如电商场景

**这一步有个关键技巧**：

> 这些小机器人发现问题时，**不是给人看报错**，而是**给 AI 看报错**——并附一句"你应该这样改"。

实习生看到反馈，会**自己改一版再交**。一来一回——**这就是 harness 真正强大的地方**。

> 技术文档里叫 **Sensor / 反馈控制 / 自纠正循环**。

---

### 第 4 件：**大事必须找你**

有些事，实习生**不许自己拍板**：

- 退款超过 1000 块 → 找经理
- 删除超过 100 个文件 → 找你
- 给客户发律师函 → 找法务
- 下单超过 5 万 → 找风控

**这条最重要也最容易出事**——不是按钮做不出来，难的是 **流程问题**：

- 在哪儿挡？
- 谁来审？
- 多久必须审完？
- 没人审，是默认放行还是默认拒绝？

先用最粗暴的"卡死等审批"跑通就行，再优化成采样人审。

> 技术文档里叫 **HITL（Human-in-the-Loop，人在回路）**。

---

### 第 5 件：**他做的每件事都记账**

实习生做的每件事都要写进日志：

- 几点几分接的活
- 看了哪些资料
- 用了什么工具
- 中间想了什么
- 最后输出了什么
- 花了多少 token、多少钱

**为什么重要？**

- **出事好复盘**——回放整个过程，找出哪一步错了
- **看 AI 的"成长"**——慢慢知道它哪里能力强、哪里弱
- **给"小机器人"提供数据**——没记录就没法度量

> ⚠️ **这件事是 5 件事里最先做的**——没有日志，后面 4 件事都做不好。  
> 技术文档里叫 **Observability / Trace / 可观测性**。

## 三、它干活的一天

把 5 件事串起来，实习生（AI）干一件活的过程是这样：

1. **接活**（"帮我处理这个用户工单"）
2. **想一下**：根据《工作手册》和上下文，制定一个步骤
3. **干一步**：调一个工具（比如查订单）
4. **小机器人检一下**：参数对不对、有没有越权
5. **看结果**：这一步出来什么数据
6. **再想下一步**：根据新结果，决定下一步
7. **遇到大事**：暂停，等人审批
8. **干完了**：把整件事的产出再给小机器人审一遍
9. **检不过就改**：根据小机器人反馈，自己改一版，再交
10. **改不动了就求救**：连续 3 次还过不了 → 升级给人

整个过程**全程被记账**，事后能完整回放。

> 技术文档里叫 **Plan → Act → Verify → Repair → Commit 循环**。  
> 翻译过来就是 **想 → 做 → 查 → 改 → 交**。

## 四、最妙的一点：管得松紧**自动调节**

这是这套框架真正高级的地方——**它会随着 AI 变强，自己放手。**

想象一个**调节面板**，每件事都有 4 档：

| 档位 | 含义 |
|---|---|
| 🚫 必须管住 | 违反就直接拦下 |
| ⚠️ 提醒一下 | 发个警告，但放过去 |
| 💡 建议 | 给个提示，AI 自己决定 |
| ⭕ 不管 | 完全交给 AI |

什么时候用哪档？看 4 件事：

| 看什么 | 调节方向 |
|---|---|
| 模型变强了（换了更聪明的） | 各项管控 → **放松** |
| 任务变难、风险变高 | 各项管控 → **收紧** |
| 业务对错误特别敏感（金融、医疗） | 默认 → **更严** |
| 业务可逆性高（改代码可 revert） | 默认 → **更松** |

**这件事用一个配置文件控制，不用改代码。**

实习生今年新人 → 全开 🚫；明年熟练了 → 部分降到 ⚠️；后年靠谱了 → 大部分降到 💡。**整个 harness 跟着 AI 一起"长大"。**

> 技术文档里叫 **Capability Profile / 能力档位**。

## 五、不同业务里这 5 件事长什么样

同一个框架，搬到不同业务就是**换"插件"**。骨架不动，里面的内容全换。

下面 5 个例子，**对照着看**会非常清楚。

### 1️⃣ 写代码 Agent（最容易做的场景）

| 5 件事 | 长什么样 |
|---|---|
| **工作手册** | `AGENTS.md`、依赖规则、PR 模板、commit 规范 |
| **工具** | 改文件、git、跑测试、CI；**禁止 force push 主分支** |
| **小机器人** | 编译器、lint、单测、覆盖率、安全扫描——**这些都是免费的！** |
| **找人审** | 改 infra、跨包大改、合主分支 |
| **记账** | git 历史 + agent 操作日志 |

**为什么最容易？** 写代码场景**小机器人现成又便宜**（编译器/测试），改错了还能 revert。

---

### 2️⃣ 客服 / 工单 Agent

| 5 件事 | 长什么样 |
|---|---|
| **工作手册** | 品牌话术、不能承诺的事项清单、对话流程 |
| **工具** | CRM 查询、发邮件、退款、改地址；**金额 > 阈值不给** |
| **小机器人** | 敏感词、PII 泄漏检测、违禁承诺检测、退款金额校验、用 LLM 评"语气是否合适" |
| **找人审** | 高金额退款、VIP、二次升级、舆情风险 |
| **记账** | 对话全文 + 调用接口 + 用户反馈 |

**特点**：文字一发出去就**收不回来**（用户看到了），所以邮件先沙箱预览。

---

### 3️⃣ 数据分析 / 报表 Agent

| 5 件事 | 长什么样 |
|---|---|
| **工作手册** | 表 schema、**指标口径文档**、查询模板 |
| **工具** | 跑 SQL、建看板、Notebook；**生产库默认只读** |
| **小机器人** | SQL dry-run、行数上限、`SELECT *` 拦截、PII 列审计、用 LLM 评"SQL 是否回答了原问题" |
| **找人审** | 写库、跨域 join、对外报表、营收类指标 |
| **记账** | 每条 SQL + 结果哈希 + 谁问的 |

**特点**：错了**没人立刻发现**（指标看上去对，但口径错了）——所以"小机器人"的判断特别重要。

---

### 4️⃣ 金融 / 交易 Agent

| 5 件事 | 长什么样 |
|---|---|
| **工作手册** | 策略边界、监管约束、限额表、可交易品种白名单 |
| **工具** | 行情、下单、撤单、对账；**新策略只许影子运行** |
| **小机器人** | 仓位限额、滑点上限、合规黑名单、异常波动熔断 |
| **找人审** | 超阈值订单、新策略首次上线、异常市场 |
| **记账** | **监管要求 100% 留痕** |

**特点**：**钱出去就回不来**——所以默认档位是 🚫；下单前要双重校验、要熔断器、要 kill-switch。

---

### 5️⃣ 医疗 / 法律咨询 Agent

| 5 件事 | 长什么样 |
|---|---|
| **工作手册** | 专科指南、引用规范、免责声明模板、禁用绝对化措辞 |
| **工具** | 医学/法规库检索、病历/合同读取、起草模板 |
| **小机器人** | 必须给出引用、必须含免责声明、引用真实性校验、专科 LLM 复核 |
| **找人审** | **几乎全场景都需要**——AI 只做"草稿+检索"，专家签字 |
| **记账** | 法定保留时长（多年） |

**特点**：错误代价**极高**（健康/法律责任）——所以 AI 不直接面向终端用户出最终意见。

---

### 横着看 5 个场景的规律

从 **写代码 → 客服 → 数据 → 金融 → 医法**，越靠后：

| 这件事越突出 | 表现 |
|---|---|
| **不可逆性** | 从能 revert → 完全无法撤回 |
| **小机器人类型** | 从计算型（lint）→ 推理型（LLM judge）→ 必须人审 |
| **找人审频率** | 从偶尔 → 常态 → 全场景 |
| **默认档位** | 从 💡 → 🚫 |
| **错误代价** | 从"加班修一下" → "罚款/吊销执照" |

**一句话**：**业务越不可逆、错误代价越高，harness 就越是必需品。**

## 六、从零开始怎么搭？（7 步白话版）

如果你今天就要落地，按这个顺序：

| 步 | 做什么 | 为什么 | 大概多久 |
|---|---|---|---|
| **1** | **先做记账**，所有动作写日志 | 没日志，后面都白搭 | 3-5 天 |
| **2** | 装 2-3 个**最便宜**的小机器人 | 写代码：lint+测试；其他业务：先建一份"标准答案集" | 2-4 天 |
| **3** | 写第一份《工作手册》 | **先就管 1-2 条规则**，不要贪多 | 持续投入 |
| **4** | 装 2 道硬护栏 | 危险操作白名单 + 钱/token 预算上限 | 2-3 天 |
| **5** | 把上面 4 步用一个**配置文件**串起来 | 这就是"调节面板" | 1-2 周 |
| **6** | 给最危险的几件事加"找人审" | **先用最粗暴的卡死等审批**，能跑就行 | 协议比代码重要 |
| **7** | 做覆盖率检查 | 看哪些没在管，把没用的关掉 | 数月起步 |

> **粗略**：第 1-4 步大约 **2-4 周**就能跑起来；  
> 第 5-7 步要 **3-6 个月** 才能稳定。  
> **不要追求一步到位**，每一步停下来都能用。

## 七、判断这套东西**该不该做**

不是每个 AI 项目都需要。

✅ **该做**：

- 你的 AI 要做**多步骤、自主**的活（不只是回答一句话）
- 错误**会带来真实代价**（钱、用户、合规、安全）
- AI 的产出**会被真的采用**（不是 demo，是上线）
- 团队**不止一个人**用这个 AI，需要统一规范

❌ **可以先不做**：

- 只是个聊天/问答机器人，不调工具、不改东西
- 还在 demo 阶段，没有真实用户
- 任务可逆，错了重来一遍就行
- 一个人用，自己审就够

## 八、术语对照表（回头看技术文档时用）

读 [design.md](./design.md) 看到这些词，对应这里的：

| 技术文档里 | 大白话 |
|---|---|
| **Agent = Model + Harness** | AI 系统 = 模型本身 + 一整套围着它转的辅助 |
| **Capability** | 一件你想让"小机器人"管的事 |
| **Capability Profile** | 调节面板（每件事开多严） |
| **Capability Contract** | 让所有"小机器人"都用同一种格式汇报 |
| **Capability Registry** | 所有"小机器人"的目录册 |
| **Guide / 前馈控制** | 工作手册 + 干活前的引导 |
| **Sensor / 反馈控制** | 小机器人（事后检查 + 给 AI 看的反馈） |
| **计算型 Sensor** | **不需要 AI 就能跑**的小机器人（lint、规则、校验） |
| **推理型 Sensor** | **得调另一个 AI 来判断**的小机器人（LLM judge） |
| **HITL** | 找人审 |
| **Observability / Trace** | 记账 |
| **Repair Loop** | 检不过自己改的循环 |
| **Eval / Harness Coverage** | 给"小机器人们"打分的考试 |
| **Shadow Mode** | 影子运行——AI 跑但不真做 |
| **Block / Warn / Hint / Off** | 必管 / 提醒 / 建议 / 不管（4 档调节） |
| **Kernel** | 把上面这些东西串起来的"调度核心" |
| **Plan → Act → Verify → Repair → Commit** | 想 → 做 → 查 → 改 → 交 |

## 九、一句话收尾

> **harness engineering 不神秘——它就是"给一个会出错的聪明小孩配齐《工作手册》、工具、检查员、审批人和记账本"。**  
>
> **聪明的不是手册，是小孩；但能不能放心让他干活，靠的全是手册和检查员。**

读到这里你已经掌握了它的**全部核心**。  
剩下的所有技术文档，只是把这 9 条用更工程化的语言重写一遍。
