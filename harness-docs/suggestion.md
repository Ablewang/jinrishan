# 直接说结论：**不难起步，难在落地后维护**

按你这份 design.md 来评估，它的难度结构很**典型**：**头 20% 的工作量能拿到 70% 的价值**，但**最后那 20% 的「可演化」效果**会持续吃掉时间。

---

## 一、按工作量拆开看（按你 design.md 第七节的 7 步）

| 阶段 | 工作内容 | 工程难度 | 真正的难点 | 估算（小团队 1–2 人） |
|---|---|---|---|---|
| 1. Observability | 结构化 trace、可重放 | ⭐⭐ 低 | **schema 早期定不稳** —— 后面所有数据都靠它 | 3–5 天能跑 |
| 2. 计算型 Sensor 接入 | lint / type / test 串到 repair 回路 | ⭐⭐ 低 | 把报错改成「**给 LLM 看的提示**」而不是给人看 | 2–4 天 |
| 3. Guide 层 / `AGENTS.md` | 显性化团队隐规则 | ⭐⭐⭐ 中 | **共识比代码贵**：团队不愿写、规则会过期 | 持续投入 |
| 4. Safety + Cost 硬护栏 | 命令白名单、token 预算 | ⭐⭐ 低 | **沙箱**比白名单难十倍；预算告警容易被忽略 | 2–3 天到「能用」 |
| 5. Capability Profile + Registry | 配置驱动、合约接口 | ⭐⭐⭐ 中 | **接口一旦错了就全员返工**；要尽量晚再固定 | 1–2 周到 v0.1 |
| 6. HITL 门禁 | 审批 + 采样复核 | ⭐⭐⭐ 中 | **不是技术问题**：在哪儿挡？谁来审？SLA 多久？ | 协议比代码重要 |
| 7. Eval & Harness Coverage | golden set、mutation、shadow | ⭐⭐⭐⭐ 高 | **数据集才是核心资产**；建得不好 Harness 就退化成纯感性调参 | 数月起步，长期投入 |

> **粗略**：能用的 v0.1 大约 **2–4 周（1–2 人）**；能演化、能让位的 v1.0 通常要 **3–6 个月** 才会稳定。

## 二、真正会卡住你的 5 件事（按风险排序）

1. **Trace schema 早期定型**  
   后面的 Sensor、Eval、Coverage 全靠 trace。一旦字段不稳，回看就废。**第一个动作就是把 trace 字段定到团队都点头**。

2. **Sensor 信号 → "给 LLM 用的提示"**  
   现成 `tsc` / `eslint` 的输出**不是面向 agent 的**。每个 Sensor 都要写 message template，这件事**没法外包给开源**，必须自己迭代。

3. **Capability Contract 的接口稳定性**  
   你 design.md §六.1 那个 `Capability` 接口，**写完第一版后大概率会改 2–3 次**。建议先用 **3 个不同实现去验证它**（一个 lint、一个 LLM judge、一个人工审批），跑通了再发布。

4. **HITL 不是写按钮的事**  
   "在哪儿挡 / 谁审 / 多久 / 漏审怎么办" —— 这是**流程设计**，比代码贵。先用**最粗的同步阻塞**跑通，再优化为采样/异步。

5. **Harness Coverage 容易做成「只生成不消费」**  
   覆盖率指标如果没有人定期看、没有触发动作（比如关闭 idle sensor），就是花钱的摆设。**指标必须挂动作**，否则砍掉。

## 三、复杂度的「上限」由你自己设

这套设计**有意做成可裁剪的**，所以难度可以选：

- **轻量版（1–2 周可上线）**：trace + 3 个 Sensor + AGENTS.md + 一个 token 预算 + 一个危险命令白名单。**没有 Profile / Registry / Coverage**——直接硬编码。**够 80% 团队用了**。
- **中量版（1–2 月）**：加 Capability Profile + Registry + HITL 门禁；模型升级走配置而不是改码。
- **完全体（半年起）**：Eval 集 + Mutation 测试 Sensor + Shadow run + 自动让位策略——这一层是「**Harness 自己也是工程**」的真正含义，**不建议第一版做**。

## 四、你这份设计的"难度自带降配"

我看了一下 design.md，它**已经把难度做了一次软化**，这点是好的：

- §四的 `strength: block / warn / hint / off` —— 让能力**先以 hint 上线**，争议低、回滚快。
- §六.3 的 Coverage —— 即使做不全，**只要做 false-positive 率和 signal-to-fix ratio 这两个**，就能反向证明「该能力是否值得保留」。
- §七的 7 步是**严格按"独立产出价值"排过序的**——哪一步停下来都不会半成品。

> 真正"难"的部分（Eval / Coverage / 自动让位）放在了最后，这是**工程上正确的延迟**。

## 五、给你的建议

如果今天就开工，我会**先做这 4 件事**，工作量都不大但能立刻产生约束力：

1. 在 `harness-docs/` 旁边建 `harness/` 包，先写 **trace 写入 + 一个 lint sensor + 一个 token 预算**。
2. 写一份**最小 `AGENTS.md`**（比如先就管"不要跨包直接 import"这一条）。
3. 先**不做** Capability Profile，硬编码跑通整个 plan→act→verify→repair 闭环。
4. 等接入了第 2 个 Sensor 类型（比如 LLM judge），**接口痛点自然冒出来**——这时再抽 Capability Contract，会一次抽对。

**简单说**：这套架构**不在于写得多复杂，而在于敢不敢晚抽象**。先跑数据、再抽接口、再做让位策略——按你 design.md 的顺序老老实实推，**两周就能见到效果**。
