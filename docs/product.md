# 产品规划

## 一句话定位

> 今日膳是家庭膳食决策助手——在你喜欢的菜里帮你做选择，顺带解决买菜和做法的问题。

---

## 核心用户

**主要用户：家庭膳食决策者**
通常是家里负责做饭的那个人（不分性别），每天面临"今天做什么"的决策压力。特征：
- 知道家人喜欢什么、不吃什么
- 会做的菜有限，容易陷入重复
- 没时间研究复杂食谱，但也不想每天将就
- 买菜没计划，常常买多浪费或临时发现缺食材

**次要用户：家庭其他成员**
主要是消费推荐结果（查看本周菜单、提意见），不是主要操作者。

---

## 核心价值主张

**不是"给你更多选择"，而是"帮你做决定"。**

搜食谱类产品（下厨房、美食天下）解决的是"我知道想吃什么、但不知道怎么做"。
今日膳解决的是更前一步的问题——**我不知道该吃什么**。

关键差异：主动推荐（push）而非被动搜索（pull）。

---

## 痛点与对应功能

| 痛点 | 功能 |
|------|------|
| 每天不知道吃什么 | 今日推荐 |
| 老是重复那几道菜 | 多样性算法（近期吃过的降权） |
| 不知道家人喜不喜欢 | 家庭口味档案 |
| 买菜没计划、容易浪费 | 周计划 + 买菜清单 |
| 想做新菜但不会 | 菜谱详情 + 步骤引导 |
| 冰箱有食材不知道做什么 | 以食材找菜谱 |

---

## 功能优先级

### P0 — MVP，验证核心价值
- **家庭口味档案**：成员管理、喜好标签、忌口设置
- **今日推荐**：基于档案，每天推三餐，可以换一换
- **菜谱详情**：食材列表 + 分步骤图文

### P1 — 形成使用习惯
- **周计划**：七天菜单，可手动调整
- **买菜清单**：从周计划自动生成，按品类分组，可勾选
- **快捷模式**：今天懒了（20 分钟内）/ 今天有客人
- **Bot 页面**：自然语言对话入口，说"今晚吃什么"直接拿到推荐，操作反哺偏好

### P2 — 提升留存与差异化
- **以食材找菜谱**：输入冰箱现有食材，推能做的菜
- **家庭协作**：多人共享菜单，成员可以点赞/反对
- **历史记录**：吃过什么，避免近期重复
- **季节感知**：自动推应季菜

### P3 — 商业化探索
- **一键生成购物订单**（接美团买菜/叮咚买菜）
- **高级版**：多家庭、高级统计、无限历史

## 关键用户流程

### 新用户 Onboarding（目标：2 分钟完成）
```
下载打开
  → 创建家庭 + 添加成员
  → 卡片滑动建立口味档案（每人约 20 张，喜欢/不吃/跳过）
  → 看到第一次推荐
```
**关键原则：** 不填表，只滑卡片。完成前就能看到推荐，让用户感受到价值。

### 每日使用流程（目标：30 秒决策）
```
收到推送通知："今晚做什么？"
  → 打开看到今日推荐（三餐或只看晚餐）
  → 不喜欢就换一个
  → 确认后进入菜谱详情
  → 做完标记为已吃
```

### 周计划流程（每周一次，目标：5 分钟）
```
周末打开周计划
  → 系统已预填七天菜单
  → 调整不满意的
  → 确认 → 生成买菜清单
  → 去超市对照清单购买
```

---

## 设计原则

1. **给答案，不给选项** — 推荐要有态度，不是给一堆让用户自己选
2. **操作步骤越少越好** — 核心场景（看推荐、换一个、看做法）最多 3 步
3. **允许懒用户** — 不记录、不填写也能用，只是推荐质量低一些
4. **家庭感** — 界面和文案要有温度，不是冷冰冰的工具

---

## 最大风险

**推荐质量** — 如果推荐总是不准，用户会快速流失。
前期宁可减少菜谱数量，保证推荐的每一道菜都是用户真正可能喜欢的。

**习惯养成** — 产品价值需要每天触达。
关键是找到合适的推送时机（下午 5 点前后），让"打开今日膳看今晚吃什么"成为习惯动作。

---

## 系统自我升级

这是产品的长期护城河——越用越懂你，越用越准。分三个层次：

### 第一层：用户行为学习（个人维度）

不只靠用户手动填写的口味档案，更从行为里反推偏好：

- 接受了哪道推荐 → 加权该菜的相关标签
- 拒绝 / 换掉了哪道 → 降权或记录短期回避
- 某类菜总被跳过（即使没标注不吃）→ 推断隐性忌口
- 做完标记"已吃"的菜 → 纳入近期历史，近期不重复推

**结果**：用户用得越久，档案越准，推荐越贴合，不需要用户手动维护。

### 第二层：用户自建菜谱

用户可以录入自家私房菜，带完整结构化标签：

- 自建菜谱优先进入该家庭的推荐池
- 对系统来说和普通菜谱无异，同样参与推荐逻辑
- 条件成熟后，高质量用户菜谱可审核后晋升进系统库
- 系统定期新增菜谱（节令菜、流行趋势），保持菜库新鲜感

### 第三层：群体数据反哺（跨用户维度）

单个用户产生不了的洞察，靠规模来获得：

- 哪些菜在夏天被接受率更高 → 季节权重
- 哪两道菜放在同一周计划里总被接受 → 搭配规律
- 南方用户更偏清淡，北方更偏咸鲜 → 地域口味模型
- 某菜谱被大量用户拒绝 → 可能标签有误，触发数据审查

**飞轮效应**：用户越多 → 数据越丰富 → 推荐越准 → 更多用户留存。

### 实现路径

| 阶段 | 做什么 |
|------|--------|
| 早期 | 记录接受/拒绝事件，简单规则加权 |
| 中期 | 协同过滤（相似家庭喜好互相参考） |
| 后期 | 引入 AI 理解自然语言反馈、自动打标签、生成新菜谱 |

## Bot 页面设计

### 定位

Bot 不是聊天机器人，而是推荐引擎的**自然语言入口**。用户说话，Bot 理解意图，调用推荐模块，把结果以对话形式呈现。

### 用户场景

```
用户："今晚有点累，想吃点清淡的"
Bot：推荐 3 道符合条件的菜（卡片形式）+ "要不要加入今晚计划？"

用户："家里有土豆和排骨"
Bot：用食材搜索推荐 → 返回可做的菜谱

用户："最近老推川菜，换换口味"
Bot：识别偏好调整意图 → 记录到 user_preferences → 下次推荐降权川菜

用户："帮我排一下这周的菜"
Bot：生成周计划草稿 → 展示 7 天安排 → 用户确认或调整
```

### 消息类型

| 用户意图 | Bot 处理 | 返回内容 |
|----------|----------|----------|
| 今天 / 今晚吃什么 | 调用推荐引擎（当餐） | 菜谱卡片 × 3，含换一换 |
| 以食材找菜 | 食材匹配 | 菜谱卡片列表 |
| 修改偏好 | 更新 user_preferences | 确认文本 + 后续推荐变化说明 |
| 排本周菜单 | 生成 weekly_plan 草稿 | 周计划卡片，可点击调整 |
| 纯闲聊 / 无关 | LLM 兜底回复 | 引导到支持的功能 |

### 数据流

```
用户输入（自然语言）
        ↓
LLM 意图识别 + 参数提取
  - intent: recommend / ingredient_query / preference_update / weekly_plan
  - params: { meal_type, constraints, ingredients, ... }
        ↓
调用推荐引擎 / 偏好更新 API
        ↓
结果组装成消息（文本 + 交互卡片）
        ↓
用户操作（接受 / 拒绝 / 换一个 / 加入计划）
        ↓
写入 recommendation_events → 反哺推荐引擎
```

### 交互卡片

Bot 回复不是纯文本，菜谱以**内嵌卡片**形式展示，卡片上可直接操作：

```
┌─────────────────────────────┐
│  🥩 红烧肉                  │
│  家常 · 咸鲜 · 中等难度      │
│  食材：五花肉、酱油、冰糖...  │
│                             │
│  [加入今晚] [看做法] [换一个] │
└─────────────────────────────┘
```

### 关键原则

- **Bot 不替代首页推荐**，首页仍是主入口，Bot 是补充（用户有具体需求时更自然）
- **操作优先于文字**：Bot 的核心价值是让用户少打字，卡片操作替代输入
- **每次操作都是数据**：点击"换一个"= rejected 事件，点击"加入今晚"= accepted 事件

---

## 暂不做的事

- 卡路里计算（用户不需要精确数字）
- UGC 菜谱社区（不是内容平台，聚焦决策）
- 餐厅推荐（先做家庭做饭场景，外卖/外食是另一个产品）
