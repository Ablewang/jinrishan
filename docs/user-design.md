# 用户侧设计

## 整体结构

```
User（账号）
  └── 属于一个或多个 Family
        └── Family 有多个 Member（每人独立偏好档案）
              └── Family 共享一套推荐 & 周计划
```

用户是独立账号，家庭是协作单元。一个用户可以属于多个家庭（如同时管理自己小家和父母家）。

---

## 登录 & 注册

### 设计原则：先给价值，再要承诺

用户打开 App，看不到任何内容就被要求注册 → 卸载。
正确路径：**让用户先感受到推荐价值，再自然地引导注册**。

### 游客模式

首次打开，不需要注册：

```
Step 1：快速选禁忌（可跳过）
  → 常见过敏原多选：花生 / 海鲜 / 香菜 / 羊肉 / 乳制品

Step 2：选口味偏好（约 10 秒）
  → 标签多选：咸鲜 / 清淡 / 麻辣 / 酸甜 / 香浓

↓ 立刻看到推荐
```

游客限制：
- 换一换有次数上限（默认 3 次/天），用完触发注册引导
- 行为数据仅本地缓存，不跨设备
- 家庭管理 / 周计划 / 买菜清单 / Bot 页面不可用

### 注册流程

```
手机号 + OTP（< 30 秒）
  → 昵称（可跳过）
  → 游客偏好静默迁移（用户无感知）
  → 可选完善偏好（卡片滑动，可全部跳过）
  → 可选创建或加入家庭（可跳过）
  → 进入完整版
```

### 登录方式
- **手机号 + OTP**（主要）
- **Apple / Google 登录**（备选，海外 & iOS 用户）
- 不做邮箱密码

## 用户数据模型

```ts
type User = {
  id: number
  phone: string
  name: string
  avatar?: string
  created_at: string
}

type Family = {
  id: number
  name: string          // 「王家厨房」
  invite_code: string   // 6位邀请码，家庭成员用这个加入
  created_by: number    // user_id
  created_at: string
}

type FamilyMember = {
  id: number
  family_id: number
  user_id: number
  nickname: string      // 在这个家庭里的称呼：「妈妈」「老公」
  role: 'owner' | 'member'
  joined_at: string
}
```

## 偏好档案

每个家庭成员独立维护自己的偏好，分三类：

```ts
type MemberPreference = {
  member_id: number

  // 1. 喜好标签（正向，推荐时加权）
  liked_cuisines: string[]        // ['川菜', '家常菜']
  liked_flavors: string[]         // ['咸鲜', '微辣']
  liked_ingredients: string[]     // ['鸡肉', '豆腐']

  // 2. 忌口（软排除，尽量避免但不强制）
  disliked_cuisines: string[]
  disliked_flavors: string[]
  disliked_ingredients: string[]

  // 3. 过敏 / 绝对不吃（硬排除，含有该食材的菜永远不推）
  allergies: string[]             // ['花生', '海鲜']
}
```

### 三类偏好的区别

| 类型 | 例子 | 推荐处理 |
|------|------|---------|
| 喜好 | 喜欢川菜 | 加分 |
| 忌口 | 不太喜欢羊肉 | 降权，偶尔还是可以推 |
| 过敏 | 海鲜过敏 | 硬过滤，永远不出现 |

## Onboarding：完善偏好档案（注册后可选）

目标：**让用户在感受到推荐价值之后，主动愿意完善档案**。

### 触发时机

- 注册完成后，系统提示「完善档案，推荐更准」（可跳过）
- 游客模式换一换次数用完，注册后自动引导
- 首页明显入口「我的口味档案还不完整」

### 流程

```
Step 1：家里有几个人吃饭？
  → 添加成员（昵称/称呼），可先跳过

Step 2：有没有绝对不吃的？
  → 游客阶段已选的自动带入，可补充修改
  → 注册后这里要提醒：「这是安全设置，含有这些食材的菜永远不推」

Step 3：喜欢什么口味？（卡片滑动）
  → 展示 20 张代表性菜品，喜欢 / 不吃 / 跳过
  → 系统从接受的菜推断偏好标签

Step 4：完成，看到更精准的推荐
```

**关键原则：**
- 所有步骤均可跳过，之后在「口味档案」里随时补充
- 游客阶段的选择自动继承，不让用户重新填
- Step 2（禁忌）注册后要再次确认，因为这时数据会真正落库并参与过滤

## 多成员偏好聚合

家庭推荐时需要综合所有成员偏好，规则如下：

### 硬规则（任意一人触发即生效）
- 任何成员的 `allergies` → 包含该食材的菜**永远不推**
- 家庭有小孩（成员年龄标记为儿童）→ 辣度自动压到 level 1 以下

### 软规则（加权平均）
- 喜好：多人都喜欢的标签 → 高权重
- 忌口：有人忌口但其他人喜欢 → 降权但不排除，偶尔推

### 冲突处理示例

| 情况 | 处理 |
|------|------|
| A 喜欢辣，B 不吃辣 | 推荐以清淡为主，偶尔推微辣（B 忌口不是过敏） |
| 全家都喜欢川菜 | 川菜高频推荐，但保持多样性不天天推 |
| A 海鲜过敏 | 所有海鲜菜永远不出现，不管其他人喜不喜欢 |

## 家庭协作

### 成员加入
- Owner 生成 6 位邀请码，分享给家人
- 家人注册后输入邀请码加入，完成自己的 Onboarding
- 不注册也能查看（只读链接，用于分享周菜单）

### 权限设计

| 操作 | Owner | Member |
|------|-------|--------|
| 修改周计划 | ✅ | ✅ |
| 确认今日推荐 | ✅ | ✅ |
| 管理成员 | ✅ | ❌ |
| 修改自己的偏好 | ✅ | ✅ |
| 修改别人的偏好 | ✅ | ❌ |
| 解散家庭 | ✅ | ❌ |

## 用户设置页

- 个人信息（昵称、头像）

- 家庭管理（查看成员、邀请、退出）
- 通知设置（推荐提醒时间）
- 账号安全（换手机号、注销）
